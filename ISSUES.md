## 如何提交问题

1. **在 GitHub Issues 中创建 Issue**，并打上对应标签（bug / hardware / control / CAN / IMU / PID 等）
2. **在本文件的「已知问题列表」中添加条目**，包含：日期、发现者、问题描述、复现步骤、影响范围
3. **问题解决后**，将条目移至「已解决问题」，补充根因分析和解决方案
4. **关联提交**：在 commit message 中引用 Issue 编号（如 `Fixes #12`）

### Issue 模板

```markdown
### 问题描述
[清晰描述现象]

### 复现步骤

### 相关日志/截图
[粘贴 VOFA+ 波形截图、串口日志、CAN 抓包等]

### 可能原因分析
[初步判断]
```

## 调试技巧与常用工具

### 调试工具链

| 工具 | 用途 | 备注 |
|------|------|------|
| **VOFA+** | 实时变量监控/波形显示 | 使用 `gimbal_stage`、`gimbal_loop_count` 等 volatile 变量 |
| **J-Link / ST-Link** | 硬件调试、断点、单步 | Keil MDK 内置支持 |
| **逻辑分析仪** | SPI/I2C/UART/CAN 总线抓包 | 通信异常时首选 |
| **CAN 分析仪** | CAN 总线监控 | 排查 CAN 丢包/仲裁问题 |
| **串口助手** | printf 调试输出 | 注意波特率匹配 |

### VOFA+ 调试变量参考

项目中已预留 VOFA+ 监控变量（见 `application/gimbal.c`）：

| 变量 | 含义 |
|------|------|
| `gimbal_stage` | 当前任务执行阶段 (1-6) |
| `gimbal_loop_count` | 任务循环计数 |
| `gimbal_tick_delta` | 两次循环的 tick 差值 |
| `can_tx_fail_count` | CAN 发送失败计数 |
| `test_iput_angle` | PID 角度环输出 |
| `angle_diff` | 目标角度与实际角度差值 |

### 常用调试步骤

1. **确认 FreeRTOS 任务调度正常**：检查各 task 是否按预期频率运行（通过 `loop_count` / `tick_delta`）
2. **确认传感器数据正确**：IMU 数据是否更新、CAN 电机反馈是否到位
3. **逐级排查控制环**：先开环测试输出，再闭合内环，最后闭合外环
4. **记录波形对比**：用 VOFA+ 对比期望值 vs 实际值，观察超调、振荡、跟踪延迟

## 常见问题 FAQ

### CAN 通信

**Q: CAN 发送失败（`can_tx_fail_count` 持续增长）？**
- 检查 CAN 总线终端电阻（120Ω）
- 检查波特率匹配（查看 `bsp_can.c` 配置）
- 检查 CAN ID 是否冲突
- 确认电机上电且 CAN 收发器正常

**Q: CAN 邮箱满导致发送阻塞？**
- 检查 CAN 发送间隔，避免高频发送
- 考虑使用 CAN 中断发送而非轮询

### PID 控制

**Q: 角度环振荡？**
- 减小 P 增益
- 增加 D 项抑制超调
- 检查 `normalize_angle()` 是否正确处理角度跳变（±180°）

**Q: 响应过慢？**
- 增大 P 增益
- 检查 `osDelay` 是否过长导致控制频率不足
- 确认电机反馈数据更新频率

### IMU / INS

**Q: 姿态解算漂移？**
- 检查磁力计（IST8310）校准状态
- 确认加速度计/陀螺仪（BMI088）零偏已校准
- 检查 AHRS 库初始化参数

**Q: IMU 数据不更新？**
- 检查 SPI 通信是否正常（用逻辑分析仪抓波形）
- 确认 IST8310 I2C 地址正确
- 检查 DRDY 中断引脚配置

### FreeRTOS

**Q: 任务卡死/不调度？**
- 检查堆栈是否溢出（`uxTaskGetStackHighWaterMark`）
- 确认各任务 `osDelay` 释放 CPU
- 检查是否有中断中调用阻塞 API

### 编译/工具链

**Q: Keil MDK 编译报错 "Undefined symbol"？**
- 检查 `.c` 文件是否加入工程
- 检查 include path 是否包含对应头文件目录
- 确认 `.lib` 库文件已添加
